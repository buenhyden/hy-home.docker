# 02-Auth Product Requirements

> This document defines the requirements for the Identity and Access Management (IAM) and Single Sign-On (SSO) gateway system.

---

## Overview (KR)

이 문서는 `hy-home.docker` 에코시스템의 핵심 보안 기반인 `02-auth` 티어의 제품 요구사항을 정의한다. 중앙 집중식 사용자 인증(IAM), 서비스 간 싱글 사인온(SSO), 그리고 ForwardAuth 기반의 서비스 보호 체계를 구축하여 시스템 전체의 보안성과 사용자 편의성을 확보하는 것을 목표로 한다.

## Vision

Provide a seamless, secure, and centralized identity foundation that enables "zero-trust" service protection and unified user access across all internal and external services.

## Problem Statement

Internal services within the `hy-home.docker` environment often lack built-in authentication or require fragmented user databases. Manually managing credentials for each service is insecure and inefficient. A centralized system is needed to enforce a single security policy and provide a unified login experience.

## Personas

- **Persona 1 (End User)**: Wants to log in once and access all permitted services without re-authenticating.
- **Persona 2 (Developer)**: Needs a standard way to protect their service without implementing custom auth logic.
- **Persona 3 (Operator)**: Needs a central dashboard to manage users, groups, and audit access logs.

## Key Use Cases

- **STORY-01**: As a user, I want to log in via a central portal so that I can access MinIO, Keycloak, and other protected apps using one set of credentials.
- **STORY-02**: As a developer, I want to use Traefik labels to enable SSO protection for my app so that I don't have to write auth code.
- **STORY-03**: As an operator, I want to enforce Multi-Factor Authentication (MFA) for the admin group to increase system security.

## Functional Requirements

- **REQ-PRD-FUN-01**: The system must provide an OIDC/SAML compliant IAM provider (Keycloak).
- **REQ-PRD-FUN-02**: The system must implement a ForwardAuth gateway (OAuth2 Proxy) to intercept and verify requests.
- **REQ-PRD-FUN-03**: User sessions must be persistent and managed in a high-performance cache (Valkey).
- **REQ-PRD-FUN-04**: Identity data must be durable and stored in a managed database (PostgreSQL).
- **REQ-PRD-FUN-05**: Support for group-based access control (RBAC) via OIDC claims.

## Success Criteria

- **REQ-PRD-MET-01**: 100% of internal web-based services are protected by SSO.
- **REQ-PRD-MET-02**: Authentication latency (ForwardAuth check) is less than 50ms on average.

## Scope and Non-goals

- **In Scope**:
  - Centralized IAM (Keycloak).
  - SSO Gateway (OAuth2 Proxy).
  - Session and Identity persistence.
- **Out of Scope**:
  - Public social login (Google/GitHub) – unless explicitly configured as IdPs.
  - Network-level isolation (VLANs/Subnets).
- **Non-goals**:
  - Replacing application-specific fine-grained permissions (e.g., individual file access in MinIO).

## Risks, Dependencies, and Assumptions

- **Dependency**: Relies on `01-gateway` (Traefik) for ingress routing and ForwardAuth redirection.
- **Risk**: Single point of failure for all services. High availability is critical.

## AI Agent Requirements

- **Allowed Actions**: Reading realm configurations, auditing logs, managing service-level clients.
- **Disallowed Actions**: Direct modification of user passwords without human-in-the-loop.
- **Evaluation Expectation**: 100% success rate in verifying session validity.
