# Kafka KRaft Guide (05-messaging)

> Zookeeper-less Kafka Cluster Setup & Management (05-messaging)

## Overview

이 가이드는 Zookeeper 없이 작동하는 KRaft(Kafka Raft) 모드 기반의 Apache Kafka 클러스터의 설정 및 운영 방법을 설명한다. 대용량 이벤트 스트리밍 및 메시지 스키마 관리를 다룬다.

## Architecture

- **KRaft Mode**: 별도의 Zookeeper 클러스터 없이 Kafka 브로커 자체가 메타데이터를 관리하여 단순성과 가용성을 높임.
- **Voters**: 3개의 브로커 노드가 쿼럼(Quorum)을 형성하여 컨트롤러 역할을 수행함.
- **Schema Registry**: 메시지 포맷(Avro, JSON 등)의 버전을 관리하여 생산자-소비자 간의 데이터 정합성을 보장함.

## Step-by-Step Instructions

### 1. 클러스터 초기화 및 실행

```bash
cd infra/05-messaging/kafka
docker compose up -d
```

- `kafka-init` 작업이 실행되어 `infra-events`, `application-logs` 등의 기본 토픽을 생성함.

### 2. 토픽 관리

- **토픽 생성**: `kafka-init` 서비스의 `command` 리스트에 새 토픽을 추가하거나 직접 명령 실행.
  ```bash
  docker exec kafka-1 kafka-topics --bootstrap-server localhost:19092 --create --topic my-topic --partitions 3 --replication-factor 3
  ```
- **토픽 조회**: `kafbat-ui` (https://kafbat-ui.${DEFAULT_URL})를 통해 웹에서 확인 가능.

### 3. 스키마 등록

- 생산자는 Schema Registry API를 통해 새로운 스키마를 게시해야 함.
- 호환성 모드(Backward, Forward, Full)에 따라 스키마 변경 시 제약 사항이 발생할 수 있음.

## Maintenance Procedures

- **Producer/Consumer 랙 점검**: `kafka-exporter`를 통해 Prometheus/Grafana에서 지연 상태를 모니터링함.
- **로그 정합성 확인**: KRaft 메타데이터 로그의 상태를 정기적으로 체크함.

## Troubleshooting

- **Leader Not Available**: 컨트롤러 노드의 쿼럼이 깨진 경우 발생. 최소 2개의 컨트롤러 노드가 정상이어야 함.
- **Schema Validation Failed**: 등록된 스키마와 전송된 메시지가 일치하지 않는 경우. 스키마 레지스트리의 최신 버전을 확인한다.
