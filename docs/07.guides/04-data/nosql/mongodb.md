## System Architecture

The MongoDB deployment utilizes a **Replica Set** architecture to provide high availability and data redundancy.

### Component Topology

-   **Primary Node (`mongodb-rep1`)**: Handles all write operations and most read operations by default.
-   **Secondary Node (`mongodb-rep2`)**: Replicates data from the primary; can take over if the primary fails.
-   **Arbiter Node (`mongodb-arbiter`)**: Does not store data. Its sole purpose is to participate in elections to break ties and ensure a quorum.
-   **Mongo-Init**: A transient container that initializes the replica set configuration on bootstrap.

### Security Model

-   **SCRAM-SHA-256**: Default authentication mechanism.
-   **KeyFile Authentication**: Internal cluster communication is secured using a shared 756-byte key file generated by the `mongo-key-generator` service.

## Usage Guide

### Management Interface

The cluster is managed via **Mongo Express**, accessible at:
-   **URL**: `https://mongo-express.local.hy-home.tech`
-   **Authentication**: Basic Auth (see `.env` or Docker secrets).

### Direct Shell Access

To interact with the database via `mongosh`:

```bash
docker exec -it mongodb-rep1 mongosh -u root -p <password>
```

## Client Connectivity

### Connection Strings

For resilient connections, use the replica set connection format:

```text
mongodb://root:<password>@mongodb-rep1:27017,mongodb-rep2:27017/admin?replicaSet=MyReplicaSet&authSource=admin
```

## Performance Monitoring

### Exporter Configuration

The `mongodb-exporter` service (Percona) scrapes metrics from the primary node and exposes them to Prometheus.
-   **Metric Endpoint**: `http://mongodb-exporter:9216/metrics`

## Resource Management

### Volume Mapping

| Service | Internal Path | Host/Volume Mapping |
| :--- | :--- | :--- |
| `mongodb-rep1` | `/data/db` | `mongodb_data1` |
| `mongodb-rep2` | `/data/db` | `mongodb_data2` |
| `mongodb-arbiter` | `/data/db` | `mongodb_arbiter_data` |

## Dependency Mapping

-   **Upstream**: Depends on `traefik` for management UI routing.
-   **Downstream**: Application services requiring document-based storage.

## Known Limitations

-   **No Sharding**: The current deployment is a single replica set; it does not support sharding across multiple sets.
-   **Arbiter Restriction**: Do not host the arbiter on the same physical node as a data node in production to maintain election integrity.
