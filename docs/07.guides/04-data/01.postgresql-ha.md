# PostgreSQL HA Guide (04-data)

> High-Availability PostgreSQL Setup & Management (04-data)

## Overview

이 가이드는 `Patroni`와 `Etcd`를 기반으로 한 고가용성 PostgreSQL 클러스터의 설정 및 관리 방법을 설명한다. 자동 장애 조치(Failover)와 부하 분산(Load Balancing) 아키텍처를 다룬다.

## Architecture

- **Patroni**: 각 DB 노드에서 실행되며 Etcd와 통신하여 리더(Primary)를 결정한다.
- **Etcd**: 클러스터의 상태 정보와 쿼럼(Quorum)을 유지한다.
- **HAProxy (pg-router)**: 외부 요청을 리더(RW)와 레플리카(RO)로 라우팅한다.

## Step-by-Step Instructions

### 1. 클러스터 실행

```bash
cd infra/04-data/postgresql-cluster
docker compose up -d
```

- Etcd 클러스터가 먼저 초기화된 후 `pg-0`, `pg-1`, `pg-2`가 순차적으로 실행된다.

### 2. 클러스터 상태 확인

```bash
# Patroni 토폴로지 확인
docker exec pg-0 patronictl -c /home/postgres/postgres.yml list
```

- `Leader` 상태인 노드가 쓰기 가능(Read-Write) 노드임.
- 나머지 노드들은 `Replica` 상태이며 `running` 중이어야 함.

### 3. Failover 테스트 (수동)

리더 노드를 수동으로 교체하거나 장애를 시뮬레이션한다.

```bash
# 수동 Switchover 실행
docker exec -it pg-0 patronictl -c /home/postgres/postgres.yml switchover
```

- 리더가 다른 노드로 넘어가더라도 `pg-router:15432`를 통한 연결은 유지됨.

### 4. 백업 및 스냅샷

- Spilo는 내부적으로 기본 백업 기능을 제공하지만, 플랫폼 수준의 백업은 `docs/09.runbooks/04-data/README.md`를 참조한다.

## Troubleshooting

- **No Leader Elected**: Etcd 클러스터의 쿼럼이 깨졌는지 확인한다 (`docker exec etcd-1 etcdctl endpoint health`).
- **Connection Timeout**: HAProxy(pg-router)가 8008 포트로 각 노드의 Patroni API에 접근 가능한지 확인한다.
- **Storage Full**: Spilo는 디스크 여유 공간이 부족하면 강제로 스냅샷 모드로 진입할 수 있다.
