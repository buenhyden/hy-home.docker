# Analytical & Specialized Databases

> Guide for InfluxDB, ksqlDB, OpenSearch, Neo4j, and Qdrant.

## 1. Context & Objective

This layer provides specialized data processing engines for time-series, graph, streaming, and vector search workloads.

- **Time-Series**: InfluxDB
- **Streaming**: ksqlDB (Kafka SQL)
- **Search**: OpenSearch
- **Graph**: Neo4j
- **Vector**: Qdrant (AI/RAG)

## 2. Infrastructure Overview

### 2.1 Time-Series & Streaming

- **InfluxDB**: Used for performance metrics and long-term analytical tracking.
- **ksqlDB**: Real-time stream processing on top of Kafka.

### 2.2 Search & Discovery

- **OpenSearch**: Full-text search and visual observability with Dashboards.
- **Neo4j**: Native graph engine for complex relationship analysis.

### 2.3 Vector Search (AI)

- **Qdrant**: High-performance vector database for AI embeddings and RAG persistence.

## 3. Operations & Maintenance

### OpenSearch Cluster Health

```bash
curl -k -u admin:password https://opensearch:9200/_cluster/health
```

### Neo4j Browser

Accessible at `https://neo4j.${DEFAULT_URL}` for Cypher queries.

### Qdrant Snapshots

Snapshots are automated and stored in `/qdrant/storage/snapshots`.

### Search Optimization

All data objects must include `_id` and standardized timestamps.

## 4. Usage Patterns

### ksqlDB Streaming SQL

```sql
CREATE STREAM events (id VARCHAR, val INT) WITH (KAFKA_TOPIC='events', VALUE_FORMAT='JSON');
```

### Qdrant Semantic Search
Utilizes gRPC (`6334`) for high-performance vector retrieval.

## 5. Maintenance & Safety

- **JVM Heap**: OpenSearch and Neo4j require careful heap monitoring.
- **API Keys**: Qdrant access is secured via API keys in Docker secrets.
- **Logs**: ksqlDB processing logs should be monitored for stream lag.
