# NoSQL Databases (Document & Columnar)

> Guide for MongoDB, Cassandra, and CouchDB systems.

## 1. Context & Objective

NoSQL databases handle unstructured or semi-structured data at scale, providing flexible schemas and High-Availability for diverse application needs.

- **Document**: MongoDB (Replica Set), CouchDB (Cluster)
- **Wide-Column**: Cassandra (Distributed Cluster)

## 2. Infrastructure Overview

### 2.1 MongoDB (Replica Set)

A 3-node replica set for resilient document storage.

- **Services**: `mongodb-0`, `mongodb-1`, `mongodb-2`
- **Status**: Automatic leader election and sync.

### 2.2 CouchDB (Cluster)

A distributed document store optimized for sync and mobile replication.

- **Services**: `couchdb-0`, `couchdb-1`, `couchdb-2`
- **Features**: Multi-master replication and PouchDB compatibility.

### 2.3 Cassandra (Columnar)

A high-throughput wide-column store for distributed data at global scale.

- **Services**: `cassandra-0`, `cassandra-1`
- **Model**: Performance-first query-driven modeling.

## 3. Operations & Maintenance

### MongoDB Status

Check replica set health:

```bash
docker exec -it mongodb-0 mongosh --eval "rs.status()"
```

### Cassandra Status

Check cluster nodes:

```bash
docker exec -it cassandra-0 nodetool status
```

### CouchDB Initialization

Clusters must be initialized via the `couchdb-init` job or manual membership setup.

## 4. Best Practices

- **Quorum**: Use `majority` read/write concerns for consistent MongoDB/Cassandra operations.
- **Modeling**: In Cassandra, model your tables based on expected queries, not entities.
- **Resource Management**: These databases are memory-intensive; ensure adequate host resources.

## 5. Troubleshooting

- **Split Brain**: Verify node connectivity and quorum status.
- **Compaction**: Monitor Cassandra compaction cycles to avoid disk I/O bottlenecks.
- **Sync Issues**: For CouchDB, check replication logs for conflict resolution.
