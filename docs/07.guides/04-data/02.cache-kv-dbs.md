# Cache & Key-Value Stores

> Guide for managed Valkey instances and distributed clusters.

## 1. Context & Objective

Cache and Key-Value stores provide high-performance, low-latency data access for sessions, temporary states, and real-time computation.

- **Engine**: Valkey 9.0 (Open Source Redis alternative)
- **Clusters**: Distributed sharding and replication for high throughput.

## 2. Infrastructure Overview

### 2.1 Managed Instance (mng-valkey)

Standalone instance for infrastructure caching needs.

- **Service**: `mng-valkey`
- **Access**: `docker exec -it mng-valkey valkey-cli`

### 2.2 Distributed Cluster (valkey-cluster)

A 6-node cluster (3 Primaries + 3 Replicas) for application-level sharded storage.

- **Topology**: 16,384 slots distributed across 3 masters.
- **Initialization**: Managed by `valkey-cluster-init` job.

## 3. Operations & Maintenance

### Status Check

Check cluster nodes and slot distribution:

```bash
docker exec valkey-node-0 valkey-cli cluster nodes
```

### Cluster Info

Check overall health:

```bash
docker exec valkey-node-0 valkey-cli cluster info
```

### Manual Rebalance

If slots are improperly balanced:

```bash
docker exec valkey-node-0 valkey-cli --cluster rebalance localhost:6379
```

## 4. Best Practices

- **Eviction Policy**: Use `allkeys-lru` for cache-only workloads to ensure availability.
- **Persistence**: AOF (Append Only File) is enabled for durability where required.
- **Client Library**: Ensure your application supports "Cluster Mode" when connecting to `valkey-cluster`.

## 5. Troubleshooting

- **Cluster Down**: Occurs if any shard has no active master and no replica to promote.
- **MOVED Error**: Client must follow the redirection to the correct shard node.
- **Memory Pressure**: Monitor `used_memory` and `maxmemory` metrics.
