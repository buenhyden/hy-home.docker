# Relational Databases (SQL)

> Comprehensive guide for managing PostgreSQL and Supabase ecosystems.

## 1. Context & Objective

Relational databases in `hy-home.docker` provide ACID-compliant persistence for infrastructure metadata, user identities, and application workloads. We utilize both standalone managed instances and high-availability clusters.

- **Primary Engine**: PostgreSQL 17.x
- **HA Stack**: Patroni + Etcd + HAProxy (Spilo)

## 2. Infrastructure Overview

### 2.1 Managed Instance (mng-pg)

Shared PostgreSQL instance for internal tools (Keycloak, n8n, etc.).

- **Service**: `mng-pg`
- **Port**: 5432
- **Access**: `docker exec -it mng-pg psql -U admin`

### 2.2 High-Availability Cluster (pg-cluster)

A 3-node cluster with automated failover and read-only replicas.

- **Patroni**: Lifecycle management and Etcd consensus.
- **Etcd**: Distributed configuration and leader election.
- **HAProxy (pg-router)**:
  - **RW Port**: 15432 (Master)
  - **RO Port**: 15433 (Replicas)

## 3. Operations & Maintenance

### Monitoring Cluster Health

Check the Patroni topology:

```bash
docker exec pg-0 patronictl list
```

### Manual Failover (Switchover)

To perform planned maintenance on the leader:

```bash
docker exec -it pg-0 patronictl switchover
```

### Supabase Integration

Supabase provides an integrated stack (Auth, REST, Realtime) on top of PostgreSQL.

- **Studio UI**: `https://supabase.${DEFAULT_URL}`

## 4. Best Practices

- **Backups**: Standard logical dumps (pg_dump) are performed daily at 03:00 KST.
- **Security**: All connections must use TLS and credentials managed via Docker secrets.
- **Performance**: Monitor slow queries and ensure proper indexing for large datasets.

## 5. Troubleshooting

- **No Leader**: Verify Etcd health (`docker exec etcd-1 etcdctl endpoint health`).
- **RO Lag**: Check replication status in `patronictl list`.
- **Connection Issues**: Verify `pg-router` (HAProxy) is routing correctly to the active leader.
