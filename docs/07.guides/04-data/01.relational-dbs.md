# Relational Databases (04-data/relational)

> PostgreSQL 및 Supabase 에코시스템을 위한 종합 관리 가이드

## Overview (KR)

이 문서는 `hy-home.docker` 인프라의 관계형 데이터베이스(RDBMS) 계층에 대한 종합적인 기술 가이드를 프로젝트 표준에 맞춰 제공한다. ACID 준수 퍼시스턴스를 보장하기 위해 단일 인스턴스(Managed) 및 고가용성 클러스터(Patroni HA) 구성을 모두 포함하며, 데이터 정합성과 가용성을 최우선으로 관리한다.

## Guide Type

`system-guide | how-to | troubleshooting-guide`

## Target Audience

- **Developers**: 데이터베이스 연결 및 쿼리 최적화가 필요한 개발자
- **Operators**: 클러스터 상태 및 백업을 관리하는 운영자
- **AI Agents**: 인프라 구성을 분석하고 복구를 지원하는 에이전트

## Purpose

1. 관계형 데이터베이스 계층의 아키텍처(Standalone vs HA) 이해
2. Patroni 기반의 고가용성 클러스터 접속 및 관리 방법 숙달
3. 운영 정책(백업, 보안) 및 트러블슈팅 절차 체계화

## Prerequisites

- `docker` 및 `docker-compose` 설치
- `infra_net` 네트워크 접근 권한
- `patronictl` 도구 및 `psql` 클라이언트 사용 능력

## Step-by-step Instructions

### 1. Managed Instance (mng-db) 접근

`infra/04-data/operational/mng-db`에 위치한 공유 인스턴스로, 내부 도구(Keycloak 등)를 위해 사용됩니다.

```bash
docker exec -it mng-pg psql -U admin
```

### 2. High-Availability Cluster (pg-cluster) 관리

3노드 Patroni 클러스터의 상태를 확인하고 리더 전환을 수행합니다.

```bash
# 클러스터 상태 확인
docker exec pg-0 patronictl list

# 매뉴얼 스위치오버 (Planned Maintenance)
docker exec -it pg-0 patronictl switchover
```

### 3. HAProxy 엔드포인트 활용

애플리케이션은 반드시 `pg-router`를 통해 접속해야 합니다.

- **RW Port (Write)**: 15432 (Master 노드로 라우팅)
- **RO Port (Read)**: 15433 (Replica 노드로 라우팅)

## Common Pitfalls

- **No Leader**: etcd 상태 이상 시 리더 선출이 실패할 수 있습니다 (`etcdctl endpoint health` 확인 필요).
- **Direct Access**: 개별 노드(pg-0, pg-1 등)에 직접 접속하여 데이터를 수정하는 것은 권장되지 않습니다 (반드시 `pg-router` 사용).
- **Replication Lag**: 대량의 쓰기 작업 시 15433(Read-Only) 포트에서 데이터 시차가 발생할 수 있습니다.
