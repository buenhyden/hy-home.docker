# k6 Operations Policy

> k6 인프라 유닛을 활용한 성능 테스트 수행 및 인프라 운영 정책입니다.

---

## Overview (KR)

이 문서는 로드 테스팅 수행 시의 거버넌스, 리소스 할당 최소 기준, 그리고 성능 지표 데이터의 보안 및 보존 정책을 정의합니다.

## Target Audience

- Operator
- SRE
- Security Engineer

## Policy Goals

- **운영 서비스 보호**: 테스트 수행이 실제 서비스 사용자에게 영향을 미치지 않도록 관리.
- **데이터 보안**: 지표 전송 과정에서의 인증 및 기밀성 보장.
- **거버넌스 준수**: 승인된 시나리오와 정해진 윈도우에서의 테스팅 문화 정착.

## Operational Standards

### 1. 테스팅 수행 윈도우

- **Standard Test**: 평시 상시 수행 가능 (낮은 부하).
- **Stress/Soak Test**: 백업 및 정기 정검 시간대(02:00 ~ 05:00)에만 수행 권장.

### 2. 리소스 및 비용 관리

- 테스트 미수행 시 모든 `k6-worker` 컨테이너는 종료 상태를 유지하거나 최소 복제본(`1`)으로 설정해야 함.
- InfluxDB에 저장된 테스트 결과는 180일 보존 후 자동 삭제됨.

### 3. 보안 표준

- InfluxDB API 토큰은 절대 환경 변수 평문에 노출하지 않으며, Docker Secrets 메커니즘을 사용함.
- 외부망에서의 UI 접근은 반드시 VPN 또는 통합 인증(Authelia 등)을 경유해야 함.

## Monitoring Requirements

- **Alerting**: 테스트 실행 중 Gateway의 응답 성공률이 95% 이하로 떨어질 경우 즉시 중단 경고 발생.
- **Logging**: 모든 테스트 실행 기록(시작 시간, 시나리오명, 수행자)은 감사 로그에 기록됨.

## Policy Scope

이 정책은 관련 서비스의 운영 기준, 변경 통제, 검증 방법을 다룬다.

## Applies To

- **Systems**: 관련 Docker Compose 서비스와 문서화된 운영 자산
- **Agents**: repo-local governance를 따르는 AI agents
- **Environments**: local, development, homelab operations

## Controls

- **Required**: 변경 전 관련 README, guide, runbook 확인
- **Allowed**: 문서와 검증 절차의 in-place 보강
- **Disallowed**: secret 값 노출, 승인 없는 runtime 변경, 정책과 절차의 중복 SSoT 생성

## Exceptions

- 정책 예외는 사용자 승인과 관련 plan/task evidence가 있을 때만 허용한다.

## Verification

- 관련 repository validation script와 문서 heading audit로 준수 여부를 확인한다.

## Review Cadence

- 서비스 구성 변경 시 검토
- 문서 템플릿 변경 시 검토
- 주요 운영 정책 변경 시 검토

## AI Agent Policy Section (If Applicable)

- **Model / Prompt Change Process**: agent runtime 변경은 이 문서에서 직접 수행하지 않고 governance 문서로 분리한다.
- **Eval / Guardrail Threshold**: 문서 변경 후 관련 validation을 통과해야 한다.
- **Log / Trace Retention**: 검증 evidence는 task 문서나 대화 요약에 남긴다.
- **Safety Incident Thresholds**: secret 노출 또는 승인 없는 runtime 변경 징후가 있으면 즉시 중단한다.

### k6 Performance Testing Usage

> `hy-home.docker` 환경에서 k6(현재 Locust 구현체)를 사용한 부하 테스트 시나리오 작성 및 실행 가이드입니다.

---

#### Overview (KR)

이 문서는 플랫폼의 가용성 검증 및 성능 한계 측정을 위한 로드 테스팅 절차를 설명합니다. `infra/0Tooling/k6` 경로는 현재 Locust 엔진을 사용하여 InfluxDB 연동형 분산 테스팅 환경을 제공합니다.

#### Usage Type

`system-guide | performance-guide`

#### Target Audience

- QA Engineer
- SRE
- Performance Engineer

#### Purpose

서비스의 안정적인 동시 접속자 수 임계치를 식별하고, 특정 엔드포인트의 병목 현상을 정량적으로 분석하기 위한 도구 사용법을 안내합니다.

#### Prerequisites

- **네트워크 설정**: `infra_net` 브리지 네트워크에 대한 컨테이너 가시성.
- **Secrets**: InfluxDB 연동을 위한 `influxdb_api_token` 시크릿 확보.
- **Python 환경**: 시나리오 작성을 위한 Locust(Python) 스크립팅 지식.

#### Step-by-step Instructions

##### 1. 테스트 시나리오 정의

`infra/09-tooling/k6/locustfile.py` 파일을 수정하여 테스트 대상 URL과 사용자 동작을 정의합니다.

```python
from locust import HttpUser, task

class PlatformUser(HttpUser):
    @task
    def visit_homepage(self):
        self.client.get("/")
```

##### 2. 인프라 부팅

- 서비스 기동: `docker-compose --profile tooling up -d`
- 분산 워커 확장: `docker compose up --scale k6-worker=4 -d`

##### 3. 테스트 실행 및 분석

- 브라우저에서 `https://k6.${DEFAULT_URL}` (또는 내부 포트 `18089`) 접속.
- **Users**: 최대 동시 사용자 수 입력.
- **Spawn rate**: 초당 신규 사용자 증가 폭 입력.
- 테스트 중 InfluxDB를 통한 지표 수집 상태를 확인하십시오.

#### Common Pitfalls

- **Worker Disconnection**: 마스터 컨테이너 재동작 시 워커 컨테이너도 함께 재시작해야 세션이 올바르게 연결됩니다.
- **Resource Exhaustion**: 대규모 테스트 수행 시 Gateway 서비스의 CPU 부하가 급증하여 타 서비스에 영향을 줄 수 있으므로 주의하십시오.

### k6 Recovery Procedure

> k6 성능 테스트 인프라의 장애 상황 시나리오 및 복구 가이드입니다.

---

#### Overview (KR)

이 문서는 성능 테스트 중 마스터-워커 간의 통신 장애, 지표 전송 실패 또는 테스트로 인한 시스템 과부하 시의 비상 조치 절차를 다룹니다.

#### Target Audience

- Operator
- SRE
- Performance Engineer

#### Alerting & Detection

| Metric | Condition | Warning | Error |
| ------ | --------- | ------- | ----- |
| Worker Connectivity | Active Workers < 1 | Warning | Critical |
| API Success Rate | < 99.9% (In-Test) | Notice | Alert |
| InfluxDB Write Error | Error Log Count > 5/min | Warning | Critical |

#### Recovery Procedures

##### 1. 비상 중단 (Emergency Kill)

테스트로 인해 전체 시스템 가용성이 위협받는 경우:

```bash
docker compose -f infra/09-tooling/k6/docker-compose.yml down --timeout 10
```

##### 2. 마스터-워커 재동기화

워커가 마스터에 연결되지 않거나 UI에 인식되지 않을 때:

1. 마스터 로그 확인: `docker compose logs locust-master`
2. 워커 강제 재시작: `docker compose up -d --force-recreate k6-worker`
3. UI에서 `Status` 탭의 워커 카운트 증가 확인.

##### 3. InfluxDB 지표 유실 장애

1. InfluxDB 가용성 체크.
2. `influxdb_api_token` 시크릿 갱신 여부 확인.
3. 마스터 노드 재시작을 통한 원격 연결 세션 초기화.

##### 4. 시나리오 스크립트 오류 복구

- Git에서 최신 안정화 버전의 `locustfile.py`를 체크아웃하여 즉각 롤백.

#### Post-Mortem Tasks

- 장애 원인이 테스트 시나리오의 과도한 부하인지, 노드 리소스 부족인지 분석.
- 복구 완료 후 `Performance Testing Operations Policy`를 준수했는지 검증.

#### Procedure or Checklist

##### Checklist

- [ ] 관련 operation policy를 확인한다.
- [ ] 현재 compose/config/docs 상태를 확인한다.
- [ ] 필요한 절차를 수행한다.
- [ ] 검증 결과와 evidence를 기록한다.

##### Procedure

1. 관련 README와 operation 문서를 확인한다.
2. 작업 전 현재 상태를 기록한다.
3. 절차를 최소 변경으로 수행한다.
4. 검증 명령 또는 수동 확인을 실행한다.

#### Verification Steps

- [ ] 관련 validation script를 실행한다.
- [ ] 문서 변경이면 template/heading audit를 확인한다.
- [ ] runtime 변경이 있었다면 compose validation을 확인한다.

#### Observability and Evidence Sources

- **Signals**: command output, validation logs, service health status, documentation diff
- **Evidence to Capture**: 실행 명령, 결과 요약, 실패 시 원인과 조치

#### Safe Rollback or Recovery Procedure

- [ ] 실패한 문서 변경은 직전 diff 단위로 되돌린다.
- [ ] runtime 변경이 필요한 경우 이 런북 범위를 벗어난 별도 승인 절차로 분리한다.

#### Agent Operations (If Applicable)

- **Prompt Rollback**: 적용하지 않음
- **Model Fallback**: 적용하지 않음
- **Tool Disable / Revoke**: secret 노출 위험이 있으면 파일 열람을 중단한다.
- **Eval Re-run**: 관련 validation과 문서 audit를 재실행한다.
- **Trace Capture**: 변경 파일, 명령, 결과를 task evidence에 기록한다.
