### Airflow DAG Development Usage (07-workflow)

> Programmatic Workflows & CeleryExecutor Operations (07-workflow)

#### Overview

이 가이드는 `hy-home.docker` 플랫폼에서 Apache Airflow를 사용하여 데이터 파이프라인(DAG)을 개발하고 운영하는 방법을 설명한다.

#### Architecture

- **Scheduler**: DAG 파일을 읽고 작업 실행 시점을 결정함.
- **CeleryExecutor**: 수많은 워커(Worker) 노드에 작업을 분산하여 처리함.
- **Valkey Broker**: 스케줄러와 워커 사이의 메시지 큐 역할을 수행함.
- **PostgreSQL**: 태스크 상태 및 메타데이터 저장소.

#### Step-by-Step Instructions

##### 1. DAG 작성 표준

- 모든 DAG 파일은 `infra/07-workflow/airflow/dags/` 디렉터리에 위치해야 한다.
- **Idempotency**: 모든 작업은 여러 번 실행해도 결과가 동일해야 한다.
- **Connections**: 외부 DB나 API 접근 정보는 `Admin > Connections`에서 관리한다.

```python
from airflow import DAG
from airflow.operators.python import PythonOperator
from datetime import datetime

with DAG('my_standard_dag', start_date=datetime(2026, 1, 1), schedule_interval='@daily') as dag:
    ### Task definition
    pass
```

##### 2. 모니터링 및 디버깅

- **Web UI**: DAG의 성공/실패 여부를 시각적으로 확인.
- **Flower**: Celery 워커의 부하와 큐 상태를 모니터링 (`https://flower.${DEFAULT_URL}`).
- **Logs**: 각 태스크의 로그는 UI에서 직접 확인하거나 영구 볼륨을 통해 접근 가능.

#### Maintenance Procedures

- **Worker Scaling**: 부하가 높을 경우 `worker` 서비스의 복제본(replicas)을 늘린다.
- **DB Cleanup**: 오래된 작업 기록을 주기적으로 아카이빙하여 메타데이터 DB의 성능을 유지한다.

#### Troubleshooting

- **Task Stuck in Queued**: Valkey 브로커와의 연결 상태를 점검하거나 워커가 실행 중인지 확인한다.
- **DagBag Processing Timeout**: DAG 파일이 너무 무겁거나 외부 리소스를 직접 호출하는지 점검한다.
